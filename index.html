맥미니에 심은 AI를 디스코드로 꺼내놓는 법

토끼를 분양받았는데, 말을 걸어야 했다.

최병찬 2026.04.19 15분

이상한 일이다.

AI를 맥미니에 심어놓고 한 달쯤 지나니까, 심어놨다는 사실 자체를 잊어버리게 됩니다. 마치 베란다에 놓은 화분 같은 것이다. 물 줘야지 줘야지 하다가, 어느 날 보면 시들어 있거든요. 그러다 문득 깨닫는 겁니다. 심는 건 쉬운데, 쓰는 게 어렵다.

지난 글에서 Hermes를 심는 이야기를 썼습니다. 심어두기만 하고 한참 방치했어요. hermes chat 치면 답은 잘 했거든요. 근데 그걸 터미널 열어서 쓰고 있자니, 이게 뭐하는 짓인가 싶은 거다. 좋은 도구는 내가 도구한테 가는 게 아니라 도구가 나한테 와야 합니다. 스마트폰이 그랬고, 에어팟이 그랬잖아요.

그래서 나는 토끼를 디스코드에 앉히기로 했습니다.

제가 만든 AI는 영어 학원의 상담 봇이에요. 토끼 모양으로 설정해뒀고, 🥕 같은 이모지를 쓰고, 말끝에 "~당근!"을 붙이거든요. 원래는 다른 엔진 위에서 디스코드랑 이미 대화하고 있었는데, 그 엔진은 유지보수가 멈춘 지 오래라 Hermes로 옮기기로 한 겁니다.

토끼의 인격은 그대로 두고, 토끼가 서있는 바닥만 갈아끼우는 일이었다.

읽고 나면 이런 생각이 드실 겁니다.
"설치는 됐는데 쓸 줄을 몰랐어요. 이거 내일 바로 해볼 수 있겠는데?"

1부. 일단 Hermes부터 깐다

처음 이 글을 보는 사람도 따라 할 수 있게, 설치부터 정리합니다. 지난 글을 쓰고 나서 "실제로 어떻게 깔아요?" 물어본 사람이 7명이었거든요. 그러니까 그 7명에게 한 번에 답해주는 셈 치고 씁니다.

Step 1. 설치 스크립트 한 줄

$ curl -fsSL https://raw.githubusercontent.com/NousResearch/hermes-agent/main/scripts/install.sh | bash

Linux, macOS, WSL2 전부 먹힙니다. 맥미니에서는 2분쯤 걸리거든요. 끝나면 ~/.local/bin/hermes에 실행 파일이 생기고, ~/.hermes/ 폴더에 설정 구조가 펼쳐집니다.

쉘 다시 불러오는 걸 까먹지 말 것:

$ source ~/.zshrc

Step 2. 모델 붙이기

지난 글의 주제였던 부분이거든요. 요약하면 두 길.

옵션 A — OpenAI Codex로 우회

# 1) Codex CLI 설치 $ npm install -g @openai/codex # 2) 브라우저에서 한 번만 로그인 $ codex login # 3) 인증 파일을 맥미니로 복사 $ scp ~/.codex/auth.json @<맥미니>:~/.codex/auth.json

그리고 ~/.hermes/config.yaml의 model 섹션을 바꿉니다:

model: default: openai-codex/gpt-5.4 provider: openai-codex base_url: https://openrouter.ai/api/v1

정확히 3줄이지만, 어쨌든 3줄이라고 말하고 싶은 것이다.

옵션 B — OpenRouter API 키

$ hermes config set OPENROUTER_API_KEY sk-or-v1-...

돈이 붙지만 Keychain 우회 같은 건 할 필요 없거든요. 월 10달러쯤 쓰면 개인 용도로는 충분합니다.

Step 3. 잘 깔렸는지 확인

$ hermes chat

터미널에서 대화창이 뜨고, 인사하면 인사합니다. 여기까지 오면 1부 끝이거든요.

여기까지가 맥미니에 AI를 "심는" 일이었다. 이제부터는 심은 AI를 "꺼내놓는" 일이다.

2부. 디스코드에 봇이라는 모양으로 내어놓기

디스코드는 재밌는 공간입니다. 슬랙이 회사의 회의실 같다면, 디스코드는 동아리방 같은 구석이 있거든요. 그래서 여기에 AI 토끼 하나 들여놔도 어색하지 않습니다. 오히려 어울려요.

2-1. 디스코드에 "봇의 집" 만들기

디스코드에게 "나 봇 하나 돌릴 건데 주민등록증 좀 주세요" 얘기를 해야 합니다. 발급 창구는 여기거든요.

https://discord.com/developers/applications

New Application 눌러서 이름 짓고, Bot 탭으로 들어갑니다. Token 영역이 있고, Reset Token 버튼이 있거든요. 클릭하면 긴 문자열이 나옵니다. 이게 봇의 진짜 열쇠다. 한 번 발급받고 창 닫으면 다시 못 본다. 바로 복사해서 안전한 곳에 저장할 것.

그리고 같은 Bot 탭의 Privileged Gateway Intents 섹션에서 MESSAGE CONTENT INTENT를 켭니다. 이거 안 켜면 봇이 메시지 본문을 못 읽거든요. 디스코드가 개인정보 보호한다고 기본은 꺼놓는 겁니다.

봇 토큰은 곧 봇 그 자체다. 유출되면 남의 손에 토끼를 넘기는 것과 같다.

2-2. OAuth2 URL Generator — Scope와 Permission

봇은 스스로 서버에 들어갈 수 없거든요. 누가 초대해줘야 합니다. Developer Portal → OAuth2 → URL Generator로 가면 체크박스가 잔뜩 있어요.

첫 번째 박스: Scopes (봇이 어떤 자격으로 들어오는지)

체크항목뜻
✅bot봇으로 서버에 상주 (필수)
✅applications.commands/reset 같은 슬래시 명령어 쓸 수 있게

나머지 scope(identify, email, guilds 등)는 절대 체크하지 마세요. 저건 사용자 로그인 기능용이라, 봇 초대 URL에 들어가면 이상한 동의 창이 추가로 떠서 혼란만 줍니다.

두 번째 박스: Bot Permissions (서버 안에서 뭘 할 수 있는지)

이게 실제로 중요한 부분이거든요. 체크박스가 수십 개 있는데, 대화형 AI 봇 기준 최소 권한 딱 이것만 체크:

권한왜 필요한가
View Channels채널 존재를 봐야 메시지를 받는다
Send Messages답장 보내기 (당연)
Send Messages in Threads스레드 안에서 답하기
Read Message History과거 대화 맥락 읽기. 없으면 매 메시지마다 기억이 0이다
Add Reactions👀 이모지로 "메시지 받았음" 표시
Attach Files파일/이미지 첨부 답변
Embed Links예쁜 카드 형태 응답
Use External Emojis다른 서버의 커스텀 이모지로 반응하기
Manage Messages자기가 단 실수 메시지 수정/삭제
Manage Threads스레드 이름 변경/닫기
Create Public Threads긴 답변 스레드로 빼기

체크 다 하고 내리면 Generator가 밑에 합산된 숫자를 자동으로 박아줍니다. 디스코드 권한은 비트 플래그 합이라서, 각 권한에 숫자가 있고(View=1024, Send=2048, History=65536 등) 체크한 것들 다 더한 값이 이 숫자거든요. 외울 필요는 없지만 어떤 권한이 들어있는지는 알고 있어야 문제 생겼을 때 원인 찾는 겁니다.

2-3. 절대 주지 말아야 할 권한

권한왜 위험한가
Administrator전체 서버 관리자. 봇이 탈취되거나 버그 나면 서버가 삭제될 수 있다. AI 버그 하나 때문에 2년 치 커뮤니티를 날릴 수 있는 것이다
Kick/Ban Members멤버 추방. AI가 자동으로 쓸 일 없음
Manage Channels채널 생성/삭제. 실수로 채널 날릴 수 있음
Manage Server서버 이름/아이콘 변경. AI 봇이 건드릴 일 없음
Manage Roles역할 부여/회수. 권한 상승 경로가 됨
Manage Webhooks외부 연동 조작. 다른 봇 우회 경로가 됨

편하다고 Administrator에 체크하지 말 것. 스택오버플로우나 블로그에서 "그냥 Administrator 주면 된다"는 글 많은데, 그건 개발자 귀찮아서 쓴 조언이지 운영 관점에선 자살 버튼이거든요.

권한은 "필요 최소한"이 원칙이다. 부족하면 나중에 더 주면 된다. 너무 많이 주면 회수하기 어렵다.

2-4. 초대 URL을 클릭한 이후의 흐름

Generator 밑에 URL이 하나 만들어집니다. 이런 모양이거든요:

https://discord.com/oauth2/authorize?client_id=<봇_ID>&permissions=<숫자>&scope=bot+applications.commands

이 URL을 로그인된 브라우저에서 클릭하면 디스코드가 순서대로 창을 띄워요.

① 서버 선택 화면

"봇 이름을(를) 어느 서버에 추가하시겠습니까?" 드롭다운이 뜨는데, 본인이 "서버 관리" 권한이 있는 서버만 여기 나옵니다. 멤버 자격만 있는 서버는 안 뜨거든요.

② 권한 확인 화면

"봇 이름에게 다음 권한을 허용합니다:" 뜨고, 아까 체크한 것들이 전부 나열돼요. 여기서 각 권한을 체크박스로 수정 가능합니다. URL에 넣은 건 기본값일 뿐, 최종 결정은 이 화면에서 하거든요.

③ "이메일 인증 필요" 경고

본인 디스코드 계정의 이메일이 인증 안 돼있으면 승인 버튼이 회색으로 비활성화되고 이 경고가 뜹니다:

⚠️ 이 봇을 서버에 추가하려면 이메일 주소를 인증해야 해요.

해결법: 디스코드 설정 → 내 계정 → 이메일 옆 "확인" 버튼 → 메일함에서 링크 클릭 → 다시 돌아와서 새로고침. 1분이면 끝나는데, 이걸 모르면 한 시간을 헤맬 수도 있거든요. 저는 30분 헤맸습니다.

④ 승인 완료

로봇 확인 한 번 나오고, 끝나면 자기 서버 멤버 목록에 봇이 들어와 있어요. 처음엔 오프라인(회색)일 겁니다. Hermes gateway가 켜져야 초록색(온라인)으로 바뀌거든요.

2-5. 권한 나중에 바꾸는 법

초대 후에도 권한 조정은 가능합니다. 디스코드 서버에서:

  1. 서버 이름 우클릭 → 서버 설정 → 역할
  2. 자동 생성된 봇 역할 클릭
  3. 권한 토글 켜고/끄기
  4. 바로 적용됨 (Hermes랑 무관)

"너무 많이 줬다" 싶은 권한은 여기서 사후 회수. 좋은 백업 안전장치거든요.

2-6. Hermes에 봇 토큰 꽂기

~/.hermes/.env 파일을 엽니다. 맨 밑에 세 줄을 추가하거든요:

DISCORD_BOT_TOKEN=<2-1에서_복사한_토큰> DISCORD_REQUIRE_MENTION=true DISCORD_AUTO_THREAD=false

각 줄의 의미:

  • 토큰: 봇의 주민등록증. 이게 전부입니다
  • require_mention: @봇이름 하고 부를 때만 반응. 안 이러면 봇이 채널 모든 대화에 끼어들거든요. 눈치 없는 동거인이 되는 거다
  • auto_thread: 답할 때마다 스레드 새로 파지 마라. 안 끄면 대화 한 번에 스레드 하나씩 생겨서 서버가 스레드로 도배됩니다

그리고 ~/.hermes/config.yaml에서 display 섹션을 찾아 이렇게 바꿉니다:

display: tool_progress: none interim_assistant_messages: false show_reasoning: false streaming: true

이걸 왜 하냐면, 기본 설정으로 두면 봇이 대답하기 전에 자기 속을 다 보여주거든요. 디스코드 채팅창에 이런 게 주르륵 뜹니다:

📖 read_file: ~/.hermes/memories/knowledge-base/con... 📖 read_file: ~/.hermes/memories/knowledge-base/ops... 🔎 search_files: "예약" 🔎 search_files: "문의|상담"

이게 개발자한테는 재밌는 정보지만, 일반 사용자한테는 공포영화 스크립트처럼 보여요. 그래서 끕니다.

2-7. Gateway 켜기

$ hermes gateway start $ hermes gateway status ✓ Gateway service is loaded

10초쯤 기다리고 상태를 보면 "loaded + PID" 나옵니다. 로그는 여기 쌓이거든요:

$ tail -f ~/.hermes/logs/gateway.log

실시간으로 뭔가 움직이는 게 보여요. 잠자던 고양이가 한쪽 눈 뜬 것처럼 깜빡이는 숫자들입니다.

3부. 토끼가 말을 안 할 때

여기까지 했으면 디스코드 서버 멤버 목록에서 봇이 초록색(온라인)으로 뜹니다. 신기해서 바로 말 걸어봐요.

나: 안녕 봇: ...

아무 말도 없습니다. 이상해요. 온라인인데 왜 답이 없지? 로그를 열어봅니다.

WARNING gateway.run: Unauthorized user: <내_ID> (<내_이름>) on discord

Unauthorized. 허가되지 않은 사용자.

내가 내 봇한테 내가 허가되지 않은 사용자라는 것이다.

이걸 pairing이라고 합니다. Hermes의 철학인데, 봇이 아무한테나 답하면 스팸이 되고 비용이 들고 남용되거든요. 그래서 기본적으로 승인된 사람만 대화할 수 있게 돼있습니다. 일종의 비밀 악수 같은 것이다.

pairing 하는 법

  1. 사용자가 디스코드에서 봇에게 DM 한 번 보낸다 (내용은 뭐든 상관없음)
  2. 봇이 이렇게 답장한다:
    Hi~ I don't recognize you yet!
    Here's your pairing code: XXXXXXXX
    Ask the bot owner to run:
    hermes pairing approve discord XXXXXXXX
  3. 운영자가 맥미니 터미널에서 시키는 대로 친다:
    $ hermes pairing approve discord XXXXXXXX
  4. 끝. 이제부터 그 사용자는 모든 서버에서 이 봇과 대화할 수 있다.

pairing은 서버별이 아니라 사용자별입니다. 한 번 승인하면 A 서버에서도, B 서버에서도, DM에서도 다 되거든요.

pairing은 번거롭지만 한 번만 하면 된다. 봉인을 푸는 의식 같은 것이다.

4부. 우리가 걸린 함정 네 가지

일이 이게 끝이면 좋았을 텐데, 그렇지 않았습니다. 실전은 늘 뒷장이 더 두껍거든요.

함정 1. "분명히 승인했는데 왜 여전히 무응답?"

알고 보니 디스코드 계정이 두 개였어요. 하나는 평소 쓰는 닉네임 계정, 하나는 실명으로 만든 테스트 계정. 승인은 첫 번째에만 했는데, 테스트할 때는 두 번째 계정으로 들어가서 말을 걸었던 겁니다.

해결 방법은 두 가지:

  • 두 번째 계정으로도 DM → 코드 받기 → 승인
  • 또는 ~/.hermes/pairing/discord-approved.json에 사용자 ID를 직접 추가
$ python3 <<'EOF' import json, time, os path = os.path.expanduser("~/.hermes/pairing/discord-approved.json") with open(path) as f: d = json.load(f) d["<사용자_ID>"] = {"user_name": "<이름>", "approved_at": time.time()} with open(path, "w") as f: json.dump(d, f, indent=2, ensure_ascii=False) EOF $ hermes gateway restart # 반드시 재시작해야 반영

후자가 빠른데, 재시작 꼭 해야 반영됩니다. 이거 안 하면 파일만 고치고 왜 안 되냐고 30분 더 헤매게 되거든요. 저는 헤맸습니다.

함정 2. "갑자기 connection timeout"

첫 실행 때 로그에 이런 게 뜹니다:

ERROR gateway.platforms.discord: [Discord] Timeout waiting for connection to Discord WARNING gateway.run: ✗ discord failed to connect

심장이 내려앉아요. 근데 30초쯤 기다리면 자동으로 재시도하고, 두 번째에 붙습니다. 이게 Hermes의 설계거든요. 실패를 부끄러워하지 않고 조용히 다시 시도하는 것.

원인은 스킬을 많이 로드해놨을 때 각 스킬이 디스코드 slash command를 등록하려고 해서 그 작업이 30초를 넘기기 때문입니다. 디스코드는 한 봇당 slash command 100개 제한이 있는데, 초과분은 등록 안 되고 경고만 찍혀요. 경고지 에러가 아니거든요.

함정 3. "왜 영어로 답하지"

한국어 페르소나로 만든 봇인데 처음 몇 번은 영어로 답했습니다.

이유는 단순하거든요. 제가 영어로 말을 걸었기 때문입니다. AI는 상대 언어를 따라가는 성질이 있어서, hi라고 하면 Hi! How can I help?가 돌아옵니다. 안녕이라고 하면 한국어가 돌아오고요.

언어는 대화의 첫 문장이 결정하는 것이다. 이걸 깨닫고 나니까 억울하지 않았습니다.

함정 4. "대답 앞에 이상한 시스템 메시지"

봇이 처음 말 걸 때 이런 걸 뱉거든요:

📬 No home channel is set for Discord. A home channel is where Hermes delivers cron job results and cross-platform messages.
Type /sethome to make this chat your home channel, or ignore to skip.

영어입니다. 갑자기 영어예요. 여기서 멘붕이 올 수 있거든요. "내 봇이 왜 영어를 하지?"

이건 Hermes의 시스템 온보딩 메시지입니다. 페르소나와 무관하게 처음 한 번 뜨는 거거든요. 무시하고 다음 메시지 보내면 되고, 정말 안 뜨게 하려면 /sethome으로 홈 채널 하나 지정해주면 됩니다. cron 결과를 받을 채널 같은 걸 여기로 설정하는 것이다.

5부. 어쨌든

이제 디스코드 구석에 토끼 한 마리가 살고 있습니다. 누가 부르면 대답하고, 아니면 조용히 있거든요. 말대꾸는 안 합니다. 화도 안 내고요. 누가 밤 12시에 "내일 일정 알려줘" 하면 연결된 시스템을 찔러서 답을 가져옵니다.

근데 생각해보면, 이 모든 구조가 되게 우아해요.

디스코드 입장에서는 그냥 봇 하나 더 들어온 것이고, Hermes 입장에서는 또 하나의 플랫폼을 연결한 것이고, 제 입장에서는 토끼한테 채팅창을 준 것일 뿐이거든요. 각자의 역할이 분명하고, 서로 간섭하지 않습니다. 좋은 협업은 그래야 하는 것 같아요.

설치 자체는 총 15분쯤 걸립니다. 사용자 pairing 시간까지 합쳐도 20분. 처음 하는 사람은 함정들 때문에 2시간이 될 수도 있지만, 이 글을 따라 하면 40분이거든요. 이 40분이 "미래의 당신이 매일 여러 번 대화할 동거인"을 만드는 시간이라고 생각하면 나쁘지 않은 투자입니다.

지난번엔 Keychain이라는 금고의 벽 앞에서 7번 실패하고 호텔을 바꿨습니다. 이번엔 그 AI를 디스코드라는 채팅방에 앉히면서 4가지 함정에 빠졌고, 2개의 계정 사이에서 헤맸고, 30초짜리 타임아웃에 심장이 내려앉았거든요.

돌이켜 보면 이번에도 똑같았습니다. 기술적으로 어려운 일은 없었어요. 그냥 디스코드가 어떤 방식으로 돌아가는지 모르고 있었을 뿐입니다. pairing이 뭔지 몰랐고, permission이 비트 플래그라는 걸 몰랐고, 이메일 인증이 막는 벽이라는 걸 몰랐습니다.

문제는 늘 "모른다"는 것이었지 "어렵다"는 게 아니었습니다.

지금 이 순간에도 맥미니 안에서 Hermes가 조용히 숨 쉬고 있고, 디스코드 구석에는 토끼 한 마리가 초록색으로 빛나고 있거든요. 가끔 누가 @Dami 안녕 하면, 🥕 이모지랑 같이 답을 해요.

어쨌든, 나쁘지 않은 동거인이 하나 더 생긴 것이다.
